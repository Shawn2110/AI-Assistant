# PRD: Mobile Companion

**Track:** `mobile_companion_20260418`
**Status:** Planning
**Owner:** Shawn
**Date:** 2026-04-18

## Summary

Extend Mike from a desktop-only assistant to the user's phone via a MAUI companion app. Mike gains the ability to launch any installed app, reply to SMS, screen voicemail, and handle incoming calls by text or voice — on the user's behalf and only after the user has granted permission (gated by a user-set master password).

## User Story

> "I want Mike on my phone. When a message comes in, he can reply for me. When someone calls, he can answer or screen the voicemail. I can say 'Mike, open Spotify' and he opens it. The first time he touches any app, he asks me for my password — after that the app is approved."

## Goals

1. **Universal app launcher.** Mike can open any installed app on the phone ("open Spotify", "launch WhatsApp").
2. **Message handling.** Mike can read incoming SMS aloud, draft a reply, and send it after confirmation.
3. **Call handling.** Mike can answer incoming calls with a voice-AI persona or send them to an AI-screened voicemail. Voicemails are transcribed and summarized.
4. **First-use authorization.** The *first* time Mike accesses any given app or capability, the user must unlock with a master password. Once granted, the app is added to an allowlist. Denial is remembered.
5. **Works with existing backend.** Mobile app is a thin client that talks to the existing `src/server/` FastAPI backend over the `src/tunnel/` ngrok URL. All agent logic lives on the backend — the phone just provides OS-level I/O.

## Non-Goals

- Taking over as the default phone dialer or default SMS app across the OS (Android Default SMS Role is opt-in, not required for the MVP).
- End-to-end replacement of the phone UI — Mike is an assistant, not a launcher replacement.
- iOS parity in the MVP. iOS is Phase 4 (stretch). MVP targets Android only.
- Multi-user / family accounts. One user, one device, one password.
- Off-device cloud storage of voicemails or SMS transcripts. Everything stays on the phone or on the user's self-hosted backend.

## Functional Requirements

### F1 — App Launch

- F1.1 The app enumerates all installed apps on the device (Android: `PackageManager.getInstalledApplications`).
- F1.2 The backend agent gets a new `launch_app(app_name)` tool that dispatches an RPC to the paired mobile app.
- F1.3 On the phone, the companion launches the target app's main intent (`getLaunchIntentForPackage`).
- F1.4 If the app is not in the permission allowlist, the companion blocks launch and prompts for the master password before executing.

### F2 — Messaging

- F2.1 The app registers as an SMS observer (Android `BroadcastReceiver` on `SMS_RECEIVED` + `READ_SMS` runtime permission).
- F2.2 Inbound SMS is forwarded to the backend. The agent drafts a contextual reply, returns it, the companion displays it for the user to confirm/edit, and sends via `SmsManager.sendTextMessage`.
- F2.3 In hands-free / voice mode, Mike reads the incoming SMS aloud over TTS and offers to reply by voice dictation.
- F2.4 Same flow for WhatsApp if WhatsApp is present (via Notification Listener Service + Direct Reply action — no API key required). Scope for MVP: SMS only. WhatsApp is Phase 3.

### F3 — Call Handling

- F3.1 The companion uses `ConnectionService` + `CallScreeningService` (Android 10+) to gain visibility into incoming calls.
- F3.2 User-configurable modes, per caller or globally:
  - **Answer with AI**: Mike picks up, speaks a greeting, converses via the existing voice pipeline, returns a summary.
  - **Screen to voicemail**: Call is sent to voicemail. The audio (via `VoicemailContract`) is transcribed by the existing `src/voice/stt.py` and summarized by the agent.
  - **Silent ignore**: Default, no-op.
- F3.3 Transcripts and summaries are sent to the backend for agent reasoning (e.g., "Dave called, he wants to push the meeting to 3pm").

### F4 — Permission / Password Gate

- F4.1 On first install, the user sets a master password. Stored as Argon2id hash in Android Keystore (`MasterKey` + `EncryptedSharedPreferences`). Raw password never persisted.
- F4.2 An allowlist records every (app_id, capability) pair the user has approved. Approvals never expire but are revocable from the settings screen.
- F4.3 First attempt to use an un-approved app or capability triggers a password prompt on the phone. Three wrong attempts → 60-second lockout.
- F4.4 The backend never holds the password. The mobile app verifies locally and then calls the backend with a short-lived signed capability token.

### F5 — Backend Integration

- F5.1 New FastAPI endpoints: `POST /mobile/pair`, `POST /mobile/cmd`, `WS /mobile/events`.
- F5.2 Pairing uses a QR code generated by the backend and scanned by the phone. Phone and backend exchange public keys; every request afterwards is Ed25519-signed.
- F5.3 All new phone-capability tools (`launch_app`, `send_sms`, `screen_call`, `read_voicemail`) route through the mobile RPC channel rather than executing on the backend host.

## Security Requirements

- **S1.** Master password is the *only* thing gating first-time capability grants. No remote unlock.
- **S2.** All phone ↔ backend traffic is TLS (ngrok) + request-signed.
- **S3.** Mobile app refuses to execute any command whose signature fails, even if the transport is compromised.
- **S4.** Permission allowlist is stored encrypted at rest. Revocation is immediate.
- **S5.** Voicemails and SMS transcripts default to on-device storage with a 30-day retention window; uploading to the backend is opt-in per capability.
- **S6.** No telemetry. No third-party SDK beyond MAUI framework deps.

## Success Metrics

- Launch latency (`"open X"` → app visible) under 2 s on a mid-range Android device.
- SMS reply round-trip (inbound SMS → AI draft shown to user) under 4 s on LTE.
- Zero unauthorized app launches in adversarial testing (wrong password, revoked token, replay attacks).
- User can complete the full "set up password → pair with backend → approve Spotify launch" flow in under 90 seconds.

## Assumptions (Flag for User Review)

These were chosen as sensible defaults — override any that don't match your intent:

1. **Platform: Android first, iOS later.** iOS heavily restricts app launch, SMS, and call interception. Android via Intents/ConnectionService is the only realistic MVP.
2. **Password model: one master password + per-capability allowlist.** Combines ergonomic re-use with least-privilege bookkeeping.
3. **Call handling is opt-in per caller.** Screen only unknown numbers by default; contacts pass through normally.
4. **Voicemail screening uses the existing Vosk STT on-device rather than Twilio.** Keeps the stack self-hosted.

## Open Questions

- OQ1. Should the master password also gate revocation? (Recommend: yes — symmetric.)
- OQ2. Do we want biometric unlock (fingerprint) as a secondary factor for already-approved apps, or is the allowlist-only model sufficient?
- OQ3. For call-answering, do we need true real-time bidirectional audio (`ConnectionService` with a custom audio route) or is post-call voicemail screening enough for MVP?
